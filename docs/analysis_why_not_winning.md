## 2. 「勝てない」根本原因の分析

### 2-1. 【最重大】カーブフィッティングに気づけていない

**問題:**
- `optimizer.py` のスコア関数が `PF × 勝率 × DDペナルティ` のみ
- グリッドサーチで「過去データに最もフィットしたパラメータ」を選ぶ構造
- ウォークフォワード（WF）機能は存在するが **使われているかどうかが不明**
- OOS効率（`oos_efficiency`）が1.0を大幅に下回っている可能性が高い

**証拠:**
```python
# optimizer.py:209
def _calc_score(r: BacktestResult, min_trades: int) -> float:
    pf = min(r.profit_factor, 10.0)
    wr = r.win_rate / 100.0
    dd_ratio = abs(r.max_drawdown_jpy) / max(r.total_pnl_jpy, 1.0)
    dd_penalty = 1.0 / (1.0 + dd_ratio)
    return pf * wr * dd_penalty  # ← OOS検証なしで最良パラメータを選定
```

**対策の方向:** WF の OOS 期間の成績で最終評価すべき。IS の成績だけで「勝てる手法」と判断してはいけない。

---

### 2-2. 【重大】AI シグナル（k-NN）の設計的限界

**問題:**
- `ai_learner.py` の7次元特徴量はすべて **オシレーター系**（RSI/Stoch/MACD/BB/EMA/Vol/ATR）
- k-NN は「過去の類似パターン = 未来も同じ動き」という仮定だが、FX はノンステーショナリー
- フィードバックの正解定義が `gain >= atr_mult * atr7 and gain >= 1.5 * loss` → 順張りバイアス
- `CORRECT_WEIGHT = 3.0` で正解パターンを重み付けしているが、学習データが少ないうちは過学習

```python
# ai_learner.py:192-198
if direction == 1:
    correct = gain >= threshold and gain >= 1.5 * loss  # ← 順張りのみ「正解」になりやすい
    pips    = (close[idx + forward] - entry) / pip_size
```

**対策の方向:**
- 特徴量にトレンド系（ADX, 複数タイムフレームのEMA方向, レジサポとの距離）を追加
- 正解ラベリングの条件を見直す（特にショート方向の精度が低い可能性）

---

### 2-3. 【重大】エントリースコアリングが「同点」になりやすい構造

**問題:**
- `signal_score.py` の合計は最大 115点（EMA20:20 + RSI:25 + MACD:20 + BB:15 + ATR:15 + Stoch:20）
- ただし ATR は `適正ボラ = 15点` を **両方向に同じ点数** で加算
- `diff > 5` 以上でやっと方向性が決まる → 閾値が低すぎ

```python
# signal_score.py:162-170
diff = total_long - total_short
if diff > 5:       # たった5点差でLong判定
    score = min(100, 50 + int(diff * 1.5))
    direction = "long"
```

**対策の方向:** スコア閾値を上げる（diff > 15〜20 で初めてシグナル）か、重みを根拠ある値に設定し直す。

---

### 2-4. 【重大】スプレッドと執行コストの過小評価

**問題:**
- `spread_pips: float = 0.3` がデフォルト
- USDJPY の実際のスプレッドはブローカーや時間帯によって 0.2〜1.0 pip と大幅に変動
- 夜間・ニュース時は 2〜5 pip になることも
- バックテストが終日一定スプレッドで計算 → 夜間スキャルピング戦略が実際には負ける

```python
# backtest_engine.py:38
spread_pips: float = 0.3  # ← 固定スプレッドで全時間帯を評価
```

**対策の方向:** 時間帯別スプレッドモデルを実装、またはバックテストのスプレッドを最低 0.5〜1.0 pip で設定してテスト。

---

### 2-5. 【中程度】ドテン（即時反転）ロジックの問題

**問題:**
- `execute_trades` でポジション中に逆シグナルが出ると **即時ドテン**
- 短期足（1M/5M）では頻繁に発生 → スプレッドコストが累積
- 特にスキャルピング戦略でドテンが多発すると資金が急減

```python
# backtest_engine.py:710-724
if (in_trade == "long" and prev_sig == -1) or \
   (in_trade == "short" and prev_sig == 1):
    trades.append(...)  # クローズ
    # ドテン（即時反転エントリー）
    new_dir = "long" if prev_sig == 1 else "short"
    in_trade = new_dir
```

**対策の方向:** ドテンを禁止するオプションを追加、またはドテンに最低 N バーのクールダウンを設ける。

---

### 2-6. 【中程度】バックテストの「先読みバイアス残存リスク」

**問題:**
- `generate_signals` では当足 close で判定 → 翌足 open で執行 → 先読みなし（正しい）
- しかし `ボリンジャーバンド` 戦略のロジックに注目:

```python
# backtest_engine.py:264-267
elif params.strategy == "ボリンジャーバンド":
    sig = np.where(_cross_up(close, lower), 1,   # ← close が lower を上抜けたら
          np.where(_cross_down(close, upper), -1, 0))
```

- `_cross_up(close, lower)` で close > lower AND 前足 close <= lower
- BB の lower は当足 close を含む rolling 計算 → **当足 close を使って BB を計算し、同足 close でシグナル判定している**（軽微な循環参照）

**対策の方向:** インジケーター計算に `shift(1)` を加えてシグナル判定と同足の close を使わないようにする。

---

### 2-7. 【中程度】過剰な戦略数（25戦略）とパラメータ過多

**問題:**
- `optimizer.py` に 25戦略、パラメータ組み合わせ総数が膨大
- グリッドサーチで最良を選ぶほど「過去に最適化されすぎた」戦略を選びやすい
- 戦略数 × パラメータ = データマイニングバイアス が大きい

**対策の方向:** 戦略を 3〜5 に絞り、経済的根拠があるものだけに限定する。

---

### 2-8. 【参考】現在 Claude API は何をしているか

`ai_utils.py` の Claude Haiku は以下のプロンプトを送るだけ:

```
FXトレーダー向けに {symbol} の {timeframe} 足チャートを簡潔に分析してください。
直近10本の終値: ...
直近高値: ... / 安値: ...
インジケーター: ...
100〜150字の日本語で、現在のトレンド方向・強さ・注目すべき価格水準を
簡潔にまとめてください。
```

**これは「コメント生成」のみ。実際の売買判断には一切使われていない。**

## 3. ネット上で「Claudeで勝っている人」の共通手法

### 3-1. 成功者が共通して言っていること

| 勝っている人の特徴 | 本プロジェクトの現状 |
|---|---|
| Claude は **コード実装者**として使う。戦略ロジックは人間が設計 | Claude API = コメント生成のみ。戦略は固定コード |
| 必ず **OOS（未来データ）で検証**してから使う | WF機能はあるが実際の利用状況が不明 |
| プロンプトに**数値条件を明確に**書く（曖昧な指示は×） | シグナルルールはコードに埋め込まれ、Claude への指示なし |
| **単一戦略**を深く突き詰める（複数戦略は×） | 25戦略からグリッドサーチで自動選択 |
| 勝率より **期待値（EV）とRR比**を重視 | スコア関数が PF×勝率 中心 |
| **バックテスト → 最適化 → WF → フォワードテスト** の順番を守る | 最適化 → そのまま使っている可能性 |

### 3-2. 具体的に有効だった手法（ネット調査結果）

#### A. ReAct エージェント方式（Claude を自律エージェントとして使う）
```
Claude が:
1. 4つの仮説戦略を自律的に生成
2. 各戦略をバックテスト
3. 最良を遺伝アルゴリズムで最適化（Sortino比率 + 総利益を目的関数）
4. 複数期間でOOS検証
→ $10,000 → $32,000（vs buy-and-hold $16,600）
```

#### B. Pine Script 生成 + TradingView でのWFサイクル
```
Claude に戦略ロジックを詳細プロンプトで渡す
→ Pine Script v5 コードを生成
→ TradingView でパラメータを変数化してグリッドサーチ
→ OOS期間で再検証
→ 月次でパラメータを再最適化
```

#### C. プロンプト競争法（Profit Factor 2 → 5 に改善した例）
```
「ライバルのAIより良い結果が出ないと思うけど、できる？」
という競争意識を植え付けるプロンプトで出力品質が改善
```

#### D. リアルタイム LLM × MT5 ループ
```
MT5 EA → Python経由 → Claude/GPT API → 売買方向 → MT5 執行
→ ルールベースのEAをLLMの推論で代替
```

## 4. 差分：本プロジェクトに足りないもの

### 優先度 HIGH

1. **WF を「必須」にする**
   - 最適化画面から直接バックテスト→トレードではなく、WFのOOS成績でのみ戦略採用
   - OOS効率 (oos_efficiency) >= 0.7 を採用条件にする

2. **Claude API を「戦略生成」に使う**
   - 現状: チャートコメント（100-150字）のみ
   - 改善: バックテスト結果を Claude に送り、「負け要因の分析と改善案」を取得する
   - 例: 連敗パターンの特徴量を抽出 → プロンプトで Claude に分析させる

3. **スプレッドを現実的な値に**
   - デフォルトを 0.5〜1.0 pip に変更
   - 時間帯別スプレッドモデル（東京 0.3 / ロンドン 0.4 / NY 0.5 / 閑散 1.0+）

4. **スコア閾値の引き上げ**
   - `diff > 5` → `diff > 20` 以上でシグナル
   - またはシグナルを出す前に **複数タイムフレーム一致** を条件に追加

### 優先度 MEDIUM

5. **k-NN 特徴量にトレンド情報を追加**
   - 現在: RSI/Stoch/MACD/BB/EMA/Vol/ATR（全部オシレーター系）
   - 追加候補: ADX（トレンド強度）、上位TFのEMA方向、直近スイング高安からの距離

6. **戦略数を絞る**
   - 25戦略 → 5戦略（根拠のあるもの: EMAクロス、RSI×BB、ドンチャン、夜間スカルパー、トリプル確認）
   - データマイニングバイアスを削減

7. **ドテン廃止オプション**
   - バックテストでドテン頻度を計測し、コスト超過なら廃止

### 優先度 LOW

8. **Claude API をフォワードテスト評価に使う**
   - 毎日の損益サマリーを Claude に送り、「今週の負け要因」を分析させる
   - 改善ループを Claude が管理するパイプラインに

9. **BB 戦略の循環参照修正**
   - `_cross_up(close, lower)` → `_cross_up(close.shift(1), lower.shift(1))` に変更

---

## 5. 次のステップ（優先順位順）

```
Step 1: WFのOOS結果だけで戦略を評価するフローの確立
   → ウォークフォワード画面でのみ「採用可否」を判断する運用ルールを決める
   → OOS効率 >= 0.7、OOS PF >= 1.2 を採用基準に

Step 2: スプレッドを 0.5〜1.0 pip に変更してバックテストし直す
   → 夜間スキャルピング系が実際に有利かを再確認

Step 3: Claude API の活用を「戦略診断」に拡張
   → バックテスト結果 JSON を Claude Sonnet に送り、改善提案を受け取る
   → プロンプト例（後述）

Step 4: スコア閾値引き上げとk-NN特徴量改善
   → シグナル数が減っても精度が上がれば利益改善

Step 5: 戦略を5つに絞ってWFで再検証
```

## 7. 結論

| 問題 | 深刻度 | 修正難易度 |
|---|---|---|
| OOS検証なしで戦略採用 | ★★★★★ | 低（運用フローの変更） |
| スプレッドの過小評価 | ★★★★☆ | 低（パラメータ変更） |
| Claude が戦略診断に使われていない | ★★★★☆ | 中（プロンプト設計） |
| スコア閾値が低すぎ | ★★★☆☆ | 低（パラメータ変更） |
| k-NN 特徴量にトレンド情報がない | ★★★☆☆ | 中（コード追加） |
| 25戦略によるデータマイニングバイアス | ★★★☆☆ | 低（戦略を絞る） |
| ドテンのコスト過大 | ★★☆☆☆ | 中（ロジック変更） |
| BB計算の軽微な循環参照 | ★☆☆☆☆ | 低（shift追加） |

**最優先で取り組むべきこと:**
1. ウォークフォワードのOOS成績でのみ戦略採用を判断する運用ルールを徹底
2. スプレッドをデフォルト 1.0 pip に変更してリアルに近い条件でテスト
3. Claude API を「戦略診断アシスタント」として活用するパイプラインを構築
