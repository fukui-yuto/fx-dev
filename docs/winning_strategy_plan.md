# 勝てる手法 — 洗い出しと実装計画

> 作成日: 2026-03-22
> 方針: 既存プロジェクトをリプレース可として、実際に機能することが実証されている手法のみを厳選

---

## 0. 大前提：なぜ「インジケーターを作っても勝てないか」

FX で負ける理由は戦略ロジックではなく、**検証フローの欠如** が9割。

| よくある失敗パターン | 現実 |
|---|---|
| IS（過去データ）でPF=2の手法を発見 | OOSでは PF=0.8 になる（カーブフィッティング） |
| インジケーターを増やす | ノイズが増えてシグナルの精度が下がる |
| パラメータを最適化する | データマイニングバイアスが増大する |
| Claude に戦略を作らせる | Claude はコード生成ツール。戦略の有効性は人間が検証する必要がある |

**唯一の正解ルート:**
```
仮説設計 → バックテスト → OOS検証 → フォワードテスト → 採用/棄却
```

## 1. 実際に機能することが統計的に示されている戦略カテゴリ

### A. トレンドフォロー（最も実績が多い）

#### A-1. EMA クロス × ATR フィルター × 上位TF確認

**なぜ機能するか:**
- EMA クロスは「モメンタム移行」を捉える
- ATR フィルターで「ボラが出ているときだけエントリー」→ 閑散相場のノイズ排除
- 上位TFと同方向のみ → 逆張りトレードを自動排除

**具体的なパラメータ例（1H足・USDJPY）:**
```
EMA: 8 / 21
上位TF: 4H の EMA21 が上向き → ロングのみ
ATR(14) > ATR(14)の20期間平均 × 1.0 → エントリー可
SL: エントリーバーの安値（ロング）またはATR × 1.5
TP: SL距離 × 2.0（RR = 1:2 固定）
```

**バックテスト検証基準:**
- 期間: 2020-01 〜 2023-12（IS） + 2024-01 〜 2024-12（OOS）
- 最低トレード数: 100以上
- OOS PF >= 1.2、OOS 勝率 >= 40%（RR=2なら勝率40%で損益トントン）

---

#### A-2. ドンチャンブレイクアウト（タートルズ手法の変形）

**なぜ機能するか:**
- N期間高値突破 = 「強い買い手が存在する」シグナル
- プロのトレーダーも注目するレベルのため自己実現しやすい
- ノイズブレイク排除に ATR フィルターが有効

**具体的なパラメータ例（4H足・USDJPY）:**
```
ブレイクアウト期間: 20本（約5日分）
偽ブレイク除外: ブレイク後2本以内に終値が戻れば無効
ATR条件: ATR(14) / 終値 > 0.002（= 0.2%のボラ以上）
SL: 直近10本の最安値（ロング）
TP: SL距離 × 2.5
```

---

#### A-3. 200EMA トレンド確認 + プルバックエントリー

**なぜ機能するか:**
- 200EMA は多くのプロが参照 → 自己実現する
- プルバック（押し目）でエントリーすることで有利なRRを確保
- トレンド方向のみなのでランダムエントリーよりエッジが高い

**具体的なパラメータ例（1H足）:**
```
200EMA が右肩上がり（直近5本のEMA傾き > 0）
プルバック条件: 価格が 50EMA まで戻ってきたとき
確認: RSI(14) が 40〜60 にいるとき（売られすぎでも買われすぎでもない）
SL: 直近スイングロー（または200EMAの少し下）
TP: 直近スイングハイ
```

---

### B. 逆張り（レンジ相場向け）

#### B-1. BB 2σ + RSI ダイバージェンス（精度重視型）

**なぜ機能するか:**
- BB 2σ = 統計的に価格の 95% が収まる範囲
- RSI ダイバージェンスは「勢いの衰え」を示す → 反転の信頼性が上がる
- 2つの条件の組み合わせで偽シグナルを大幅削減

**具体的なパラメータ例（1H足・USDJPY）:**
```
エントリー条件（ロング）:
  1. 現在足の終値が BB(20, 2σ) の下バンド以下
  2. RSI(14) < 30
  3. 前の直近安値より現在の安値が低いが、RSI は高い（ダイバージェンス）
SL: 直前スイングロー - ATR(14) × 0.5
TP: BB ミッドライン（= SMA20）
```

**注意:** レンジ相場専用。ADX(14) < 25 のときのみ有効化する。

---

#### B-2. ピボットポイント + ストキャスティクス反転

**なぜ機能するか:**
- ピボットは世界中の銀行トレーダーが参照する価格水準
- S1/R1 レベルでの反転は高確率
- ストキャスティクスで「実際に反転している」ことを確認してからエントリー

**具体的なパラメータ例（1H足）:**
```
エントリー条件（ロング）:
  1. 価格が S1（第1サポート）付近（± ATR × 0.5 以内）
  2. Stochastic(5,3,3) の %K が 20 以下から上向きクロス
  3. ADX(14) < 25（レンジ確認）
SL: S2（第2サポート）
TP: ピボットポイント（中心）
```

---

### C. スキャルピング（短期・精度重視）

#### C-1. 夜間 USDJPY スキャルピング（東京時間 23:00〜翌3:00 JST）

**なぜ機能するか:**
- ロンドン〜NY オーバーラップ時間帯は流動性が高い
- USDJPY は夜間に方向性が出やすい
- 短期 EMA クロス + ボリューム急増で「機関投資家の動き」を検知

**具体的なパラメータ例（5M足）:**
```
時間帯: JST 22:00〜翌 2:00（ロンドン〜NY クロス）
EMA: 5 / 13 クロス
MACD(5,13,4) ヒストグラムが正方向に増加中
ATR(10) > 過去10本ATR平均 × 0.8（最低限のボラがある）
SL: エントリーバー安値 ± 1 pip
TP: SL × 1.5（RR = 1:1.5）
最大保有時間: 30分（時間切れでクローズ）
```

## 2. 「勝てる」判断基準の定義

バックテスト・WF結果の採用基準:

| 指標 | 最低基準 | 理想 |
|---|---|---|
| OOS プロフィットファクター | >= 1.2 | >= 1.5 |
| OOS 勝率 | >= 40% | >= 50% |
| リスクリワード比 | >= 1:1.5 | >= 1:2 |
| 最大ドローダウン | <= 総利益の 30% | <= 15% |
| 最大連敗数 | <= 7 | <= 5 |
| WF OOS効率 | >= 0.6 | >= 0.8 |
| 最低トレード数（OOS） | >= 30 | >= 100 |

```
    └── フォワードテスト中の戦略のリアルタイム成績表示
```

### 3-2. 戦略を 5 つに絞る理由

```
現状: 25戦略 × グリッドサーチ = データマイニングバイアス大
改善: 5戦略 × 根拠ある固定パラメータ = 信頼できる検証結果

採用する5戦略:
1. EMAクロス × ATR × 上位TF（トレンドフォロー・主力）
2. ドンチャンブレイクアウト（強トレンド向け）
3. 200EMA プルバック（中長期トレンド向け）
4. BB × RSI 逆張り（レンジ相場向け、ADXフィルター付き）
5. 夜間スキャルピング（5M足・時間帯限定）
```

### 3-3. 上位タイムフレーム確認の実装方針

現在のシステムにない最も重要な改善点:

```python
# 追加する上位TF確認ロジック（概念コード）

def get_higher_tf_trend(symbol: str, base_tf: str) -> str:
    """
    上位タイムフレームのトレンド方向を返す。
    1M → 15M
    5M → 1H
    15M → 4H
    1H → 1D
    4H → 1W
    """
    HIGHER_TF = {"1M": "15M", "5M": "1H", "15M": "4H", "1H": "1D", "4H": "1W"}
    higher = HIGHER_TF.get(base_tf)
    if not higher:
        return "neutral"

    df = get_ohlcv(symbol, higher, count=250)
    ema200 = df["close"].ewm(span=200, adjust=False).mean()
    slope = ema200.iloc[-1] - ema200.iloc[-5]  # 5本前との差

    if slope > 0 and df["close"].iloc[-1] > ema200.iloc[-1]:
        return "up"
    elif slope < 0 and df["close"].iloc[-1] < ema200.iloc[-1]:
        return "down"
    return "neutral"

# バックテスト・シグナル生成時のフィルター
def apply_htf_filter(signal: int, htf_trend: str) -> int:
    if htf_trend == "up" and signal == -1:
        return 0  # 上位足上昇中はショートを排除
    if htf_trend == "down" and signal == 1:
        return 0  # 上位足下降中はロングを排除
    return signal
```

```python
def generate_backtest_diagnosis(result: BacktestResult) -> str:
    """バックテスト結果を Claude Sonnet に診断させる"""

    losing_trades = sorted(result.trades, key=lambda t: t.pnl_pips)[:5]
    winning_trades = sorted(result.trades, key=lambda t: t.pnl_pips, reverse=True)[:5]

    prompt = f"""
あなたはアルゴリズムトレードの専門家です。
以下のFXバックテスト結果を分析し、改善提案を3点挙げてください。

## 戦略情報
- 戦略名: {result.strategy}
- 通貨ペア: {result.symbol} / 時間足: {result.timeframe}

## 成績サマリー
- 総トレード数: {result.n_trades}
- 勝率: {result.win_rate:.1f}%
- PF: {result.profit_factor:.2f}
- 最大DD: {result.max_drawdown_jpy:,.0f}円
- 期待値: {result.expected_value_pips:.2f} pips/トレード
- RR比: {result.risk_reward_ratio:.2f}
- SLヒット: {result.sl_hit_count}回 / TPヒット: {result.tp_hit_count}回
- 平均保有バー: {result.avg_hold_bars:.1f}

## 負けトレード TOP5
{_format_trades(losing_trades)}

## 勝ちトレード TOP5
{_format_trades(winning_trades)}

## 要求
1. 最も改善効果が高い変更点（パラメータ or ロジック）
2. エントリー条件の追加・削除の提案
3. SL/TP設定の最適化案

各提案は「変更内容」「期待される効果」「実装の難易度（低/中/高）」の形式で。
"""

    client = anthropic.Anthropic()
    response = client.messages.create(
        model="claude-sonnet-4-6",  # Haiku から Sonnet に格上げ
        max_tokens=800,
        messages=[{"role": "user", "content": prompt}]
    )
    return response.content[0].text
```

## 6. 最終的なリプレース判断基準

既存コードを全リプレースすべき場合:
- 上位TF確認を既存アーキテクチャに追加するのが困難な場合
- バックテストエンジンの先読みバイアスが根本的に修正不能な場合

既存コードを改修すべき場合（推奨）:
- backtest_engine.py, optimizer.py は品質が高く、改修コストが低い
- chart.py の UI は使いやすく、維持する価値がある
- 追加すべき機能（上位TF確認、診断API）は既存コードに追加可能

**推奨: 既存コードをベースに Phase 1〜2 の改修を実施、Phase 3 で部分的なリプレースを検討**

## 7. 一番重要なことまとめ

```
勝つためにやること（優先順）:
1. WFのOOS基準（PF>=1.2）を満たす戦略だけを使う（ルール）
2. スプレッドを現実的な値（1.0pip）でテストする（設定変更）
3. 上位TFと逆行するシグナルは取らない（ロジック追加）
4. インジを増やすのをやめて戦略を5つに絞る（戦略選択）
5. バックテスト結果をClaudeに診断させて改善ループを作る（拡張）

やってはいけないこと:
- IS成績だけでパラメータを選ぶ
- 「良さそうなインジ」を追加し続ける
- スプレッド 0.3 pip でスキャルピングをテストする
- 1ヶ月以内の短期バックテストを信頼する
```
